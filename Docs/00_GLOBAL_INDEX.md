# KROK.cv: Головний Індекс Проєкту

**Коротка суть:** Система віддаленої фізичної реабілітації на базі комп'ютерного зору. Продукт аналізує біомеханіку тіла пацієнта в реальному часі без використання серверних GPU, забезпечуючи конфіденційність медичних даних через E2EE-шифрування та BYOS-архітектуру.

---

##  1. Технічний стек (Tech Stack)

### Клієнтська частина (Frontend & Wrappers)
* **Core UI:** React + TypeScript.
* **Десктоп клієнт:** Electron (Windows/macOS/Linux).
* **Мобільний клієнт:** Capacitor / Ionic (iOS/Android).

### Computer Vision (ML Module)
* **Модель:** MediaPipe Pose (WebAssembly).
* **Виконання:** Локально в браузері/клієнті через ізольовані Web Workers (щоб не блокувати Main Thread).
* **Математика:** Власні алгоритми фільтрації (EMA) та векторної алгебри для розрахунку кутів.

### Серверна частина (Signaling & Backend)
* **Фреймворк:** Python / Django.
* **База даних:** PostgreSQL.
* **Реалтайм комунікація:** Django Channels / Redis (WebSockets для Signaling).

### Інфраструктура та Мережа
* **Синхронні сесії (Live):** WebRTC (E2EE P2P відео + RTCDataChannel для телеметрії), STUN-сервери Google.
* **Асинхронні сесії (Solo):** Архітектура BYOS (Bring Your Own Storage) через Google Drive API.

##  2. Правила роботи зі сховищем (Zone of Responsibility)

Кожна папка має свою сувору зону відповідальності. Крос-лінковка між папками дозволена, але логіка не має розмазуватися.

* `01_Product/` — Зона бізнесу та дизайну. Тут лежать вимоги, флоу юзера, описи цільової аудиторії, аналіз ринку та проблеми.
* `02_Architecture/` — Технічна зона. C4 Model, секвенс-діаграми (WebRTC, Data Flow), опис модулів.
* `03_ADR/` — Лог архітектурних рішень (Architecture Decision Records).
* `04_API_Contracts/` — Контракти взаємодії (JSON-структури) між Frontend, ML-воркерами та Django-бекендом.
* `05_M_Module/` — Зона штучного інтелекту та математики. Пайплайни обробки кадрів, формули фільтрації та розрахунку кутів.
* `06_Idea/` — Зона декомпозиції (Backlog). Тут розписуються дрібні фічі, які потім збираються в чеклісти у файлі `MVP_Scope`.
* `07_PHOTOS/` — Ресурси, статичні зображення, рендери UI для презентацій.

##  3. Формати та інструменти

1.  **.md (Markdown):** Основний формат для всього тексту, інструкцій та коду. 
    * Діаграми потоків та послідовностей пишемо **виключно через Mermaid** (````mermaid`).
    * Математичні формули рендеримо через **LaTeX** (`$ $` або `$$ $$`).
2.  **.canvas (Obsidian Canvas):** ТІЛЬКИ для високорівневих візуальних схем (наприклад, загальна мапа модулів). **Заборонено** дублювати сюди великі шматки тексту з `.md` файлів, щоб уникнути розсинхрону при оновленні доки.

##  4. Що таке ADR?

**Architecture Decision Record** — це лог глобальних технічних рішень. Якщо ми приймаємо рішення, яке важко змінити в майбутньому (наприклад, обрали WebRTC замість трансляції через сервер), ми фіксуємо це тут. 

**Стандартний формат ADR:**
1.  **Контекст** (Яку проблему вирішуємо?).
2.  **Рішення** (Що саме обрали?).
3.  **Наслідки** (Плюси та мінуси цього вибору, ризики).
